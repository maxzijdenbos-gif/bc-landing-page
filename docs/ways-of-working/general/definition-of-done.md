# Definition of Done (DoD)

## Context

In order to align development effort in the frontend team and to be able to
estimate better, we need to adopt a Definition of Done that will set the
requirements for when a task is considered done in the frontend team.

## Decision

DOD is currently as follows:

- All task requirements/acceptance criteria have been met as described in the
  task
- Relevant documentation/comments provided in the code/project
  repository/project wiki page
- Relevant tests have been created or updated
- Relevant Storybook stories have been created or updated (including interaction
  tests)
- Task requirements have been tested in at least the browsers outlined by the
  NFR document. (If you don't know which these are, ask your PM), both desktop
  and mobile layouts
- Task related code changes have been reviewed by the team according to our code
  review standards
- Task changes successfully passed our automated validation and testing setup

## Consequences

Implementing a Definition of Done in the frontend team will have the following
benefits:

- Better alignment and estimation of future tasks in the frontend development
  team.
- Improved code quality and readability.
- Focus on consistent user experience across browsers.
- Increased collaboration and knowledge sharing.
- Enhanced validation and testing improves overall stability and reliability of
  the application.
